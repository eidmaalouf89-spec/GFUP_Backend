**Version:** 1.0  
**Created:** 2026-04-26 (Clean Step 4)  
**Owner:** Eid  
**Status:** FROZEN — this document is the reference for Steps 5–12. Amendments require explicit Eid decision.

This document defines the final, immutable target Input / Internal / Output contract for GF Updater v3.  
It supersedes any temporary dev-state behavior currently observable in the repo.  
Steps 5–12 implement this contract. Gate 4 validates that it is fully in force.

---

## Executive Summary

GF Updater v3 currently has mixed IO behavior inherited from the parity development phase:

- The user manually places `input/FLAT_GED.xlsx` (an internal artifact, not a user input)
- The pipeline defaults to raw-GED mode (`FLAT_GED_MODE = "raw"`)
- `data_loader.py` rebuilds the raw GED on every UI load
- Outputs are partially classified and partially named consistently
- Run artifacts exist but do not yet include Flat GED artifacts
- Deprecated paths (`bet_report_merger`, `consultant_gf_writer`) are documented but not yet cleaned

This contract defines the target state after Steps 5–12. After Gate 4, the user interacts with three folders only:

```
input/      ← user places 2–3 files here
output/     ← user reads results here
data/       ← persistent memory (never deleted casually)
```

Everything else is generated, internal, or archived.

|---|---|

| Prohibited File | Why |
|---|---|
| `input/FLAT_GED.xlsx` | Flat GED is an internal artifact. The orchestrator builds it automatically. Users must not know it exists at the filesystem level. |

## Section 2 — Internal Generated Artifacts

Internal artifacts are generated by the system and consumed by the system. Users should not need to interact with them directly in normal operation.

### 2.1 Location

```
output/intermediate/
```

This folder is owned entirely by the orchestrator. Contents are overwritten on every run.

### 2.2 Artifact Definitions

| Artifact | File | Format | Format Rationale | Generated By | Consumed By |
|---|---|---|---|---|---|
| Flat GED operational layer | `FLAT_GED.xlsx` | `.xlsx` | Multi-sheet (GED_RAW_FLAT + GED_OPERATIONS + DEBUG_TRACE). Builder produces XLSX natively. | `src/flat_ged/` via orchestrator | `stage_read_flat.py` |
| Flat GED debug instances | `DEBUG_TRACE.csv` | `.csv` | Non-ACTIVE instances are large and human-readable in CSV. Not consumed by pipeline — developer-only. | `src/flat_ged/` builder | Developer inspection only |
| Flat GED build report | `flat_ged_run_report.json` | `.json` | Structured statistics (19 fields). JSON is natural for key-value metadata. | `src/flat_ged/` builder | `run_memory` registration; logging |
| Effective responses snapshot | `effective_responses.csv` | `.csv` | See §2.3 for format decision. | `stage_report_memory.py` (Step 11) | `data_loader.py` (Step 12) |

### 2.3 Format Decision: effective_responses Snapshot

**Options considered:**
- `.parquet` — compact, typed, fast to load with pandas; requires `pyarrow` or `fastparquet` dependency
- `.csv` — human-readable, no additional dependency, universally inspectable

**Decision: `.csv`**

Rationale:
1. This is a local single-user system. Load-time difference between parquet and CSV at ~31,000 rows is negligible (<1s).
2. `pyarrow` / `fastparquet` are not currently in the dependency stack. Adding them for a single artifact is not justified.
3. CSV can be opened directly in Excel for manual inspection — an operational advantage for this project.
4. DEBUG_TRACE.csv already establishes CSV as the project's format for large intermediate inspection artifacts.

**If row counts grow past ~500,000 rows in a future phase**, reconsider parquet. That is not a current concern.

### 2.4 FLAT_GED.xlsx Sheet Contract (Not Modifiable)

The Flat GED format is frozen by `docs/FLAT_GED_CONTRACT.md` (v1.0). The three sheets are:

| Sheet | Grain | Columns | Contract |
|---|---|---|---|
| `GED_RAW_FLAT` | One row per `(numero, indice)` | 21 R_COLS | Frozen v1.0 |
| `GED_OPERATIONS` | One row per `(numero, indice, step_type, actor_clean)` | 37 O_COLS | Frozen v1.0 |
| `DEBUG_TRACE` | Non-ACTIVE instances | 23 D_COLS | Frozen v1.0 |

**Do not modify `src/flat_ged/` to change this schema.**

## Section 3 — Runtime Persistent Memory

Persistent memory databases survive across runs, pipeline restarts, and OS reboots. They must never be deleted as part of routine operations.

### 3.1 Databases

| Database | Path | Role | Rows (as of 2026-04-26) |
|---|---|---|---|
| Run artifact registry | `data/run_memory.db` | Tracks all registered run artifacts, run metadata, baseline state, stale propagation | Run 0 + all completed runs |
| Consultant report memory | `data/report_memory.db` | Persists matched consultant report answers across runs | 1,245 active rows |

### 3.2 Deletion Policy

> **NEVER delete `data/report_memory.db` or `data/run_memory.db` casually.**

- `report_memory.db`: Contains 1,245+ rows of matched consultant answers that took significant effort to build. Deleting this is operationally equivalent to losing all consultant truth history. Deletion requires explicit user decision with full awareness of consequence.
- `run_memory.db`: Contains the run artifact registry, including the Run 0 baseline pointers. Deleting this breaks all artifact lineage and baseline resolution.

### 3.3 Backup Policy

**Decision: manual backup before pipeline runs that modify schema or data aggressively.**

Recommended procedure before Steps 7–8:
```
cp data/run_memory.db data/run_memory.db.bak_YYYYMMDD
cp data/report_memory.db data/report_memory.db.bak_YYYYMMDD
```

`runs/run_0000/report_memory.db` serves as a protected snapshot baseline (not to be treated as a backup of current state).

### 3.4 Integrity Checks

The following must be executable as a sanity check before any run touching the DB schema:

```python
import sqlite3
for db in ["data/run_memory.db", "data/report_memory.db"]:
    conn = sqlite3.connect(db)
    result = conn.execute("PRAGMA integrity_check").fetchone()
    assert result[0] == "ok", f"{db} integrity check FAILED"
    conn.close()
```

This is already implemented in `scripts/repo_health_check.py`. It must remain operational.

### 3.5 Optional Future Persistent Files

| File | Condition for Admission |
|---|---|
| `data/settings.json` | Only if user-configurable settings (stale-day threshold, focus mode defaults) are externalized from code |
| `data/user_prefs.json` | Only if per-session preferences are persisted (not currently needed) |

**Decision:** Neither is admitted for Steps 5–12. Deferred until explicitly scoped.

`runs/run_0000/` is the immutable project baseline.

| Property | Value |
|---|---|
| Created | 2026-04-22 |
| Source | Fresh FULL run from `input/`, with `report_memory.db` rebuilt from `input/consultant_reports` |
| Key metrics | 6,491 docs | 31,586 responses | 4,728 GF rows | 3,221 discrepancies |

If the Run 0 baseline must be refreshed (e.g., new input snapshot), create a new designated baseline run and update `docs/VALIDATION_BASELINE.md`. The original `run_0000/` is preserved regardless.

### 5.5 Stale Lineage Rules

A run's artifacts become **stale** when:
- The raw GED input has changed since the run (new export)
- `report_memory.db` has been updated since the run (new report ingestion)

Stale runs are not deleted — they remain as historical records. The `run_memory.db` stale propagation mechanism marks them appropriately. Stale runs should not be used as the inherited GF source for a new run unless no valid non-stale run exists.

### 5.6 Retention Policy

| Scope | Policy |
|---|---|
| `runs/run_0000/` | Permanent — never purge |
| Recent runs | Keep all — runs are small (XLSX artifacts + JSON + DB snapshot) |
| Manual purge | Allowed if storage is a concern, but must go through `run_memory.db` artifact deregistration |
| `runs/` total size | Monitor; no automatic purge |

There is no automatic run purge. Manual purge must update `run_memory.db` to avoid broken artifact resolution.

**Policy: keep last 20 log files, or all logs from the last 90 days — whichever is greater.**

The `temp/` folder (if created) contains files that are safe to auto-purge on pipeline startup.  
No file in `temp/` should ever be a dependency of a registered artifact or a pipeline output.  
**A pipeline run may safely delete `temp/` contents at startup.**

## Section 8 — Naming Rules

### 8.1 Principles

| Principle | Rule |
|---|---|
| Internal system files | `SCREAMING_SNAKE_CASE.ext` (e.g., `GF_V0_CLEAN.xlsx`, `FLAT_GED.xlsx`) |
| User-visible stamped exports | Natural language with spaces and date stamp (e.g., `Tableau de suivi de visa 26_04_2026.xlsx`) |
| Date format | `DD_MM_YYYY` (e.g., `26_04_2026`) — used for stamped exports only |
| Log files | `run_NNNN.log` — lowercase, zero-padded run number |
| Run folders | `run_NNNN` — zero-padded 4 digits (e.g., `run_0001`) |
| Intermediate artifacts | Lowercase with underscores for report-type files (e.g., `flat_ged_run_report.json`, `effective_responses.csv`) |

### 8.2 Canonical File Names

These are the exact names. No variants, no suffixes, no version numbers in the filename.

| File | Canonical Name | Notes |
|---|---|---|
| Primary GF output | `GF_V0_CLEAN.xlsx` | `V0` indicates reconstruction version; not a version suffix for daily runs |
| Team export | `GF_TEAM_VERSION.xlsx` | Never rename |
| Tableau export | `Tableau de suivi de visa DD_MM_YYYY.xlsx` | Date-stamped on export |
| Flat GED | `FLAT_GED.xlsx` | Never rename |
| Flat GED build report | `flat_ged_run_report.json` | Matches builder output name |
| Debug trace | `DEBUG_TRACE.csv` | Matches builder output name |
| Effective responses | `effective_responses.csv` | New (Step 11) |
| Discrepancy report | `DISCREPANCY_REPORT.xlsx` | Never rename |
| Discrepancy review | `DISCREPANCY_REVIEW_REQUIRED.xlsx` | Never rename |
| Anomaly report | `ANOMALY_REPORT.xlsx` | Never rename |

### 8.3 "Latest" Aliases

There are no symlinks or alias files. "Latest" always means the file at the canonical path (e.g., `output/GF_V0_CLEAN.xlsx`). The run artifact in `runs/run_NNNN/` is the immutable point-in-time snapshot.

### 8.4 Forbidden Naming Patterns

| Pattern | Example | Why Forbidden |
|---|---|---|
| Dates in system output names | `GF_V0_CLEAN_26_04_2026.xlsx` | Use run folder for point-in-time; canonical path is always latest |
| `_v2`, `_new`, `_final` suffixes | `FLAT_GED_v2.xlsx` | Version is tracked in `run_memory.db`, not in filename |
| Spaces in system file names | `GF V0 CLEAN.xlsx` | Breaks path handling; use underscores |
| Lowercase for core outputs | `gf_v0_clean.xlsx` | Breaks `paths.py` constants |

### 9.2 GF_TEAM_VERSION Chain (Protected)

The following chain must execute in order and must not be short-circuited:

```
stage_write_gf
  → writes output/GF_V0_CLEAN.xlsx

team_version_builder.py
  → reads GF_V0_CLEAN.xlsx
  → writes output/GF_TEAM_VERSION.xlsx

stage_finalize_run.py
  → registers GF_TEAM_VERSION artifact in run_memory.db
  → copies artifacts to runs/run_NNNN/

app.py → export_team_version()
  → reads GF_TEAM_VERSION artifact path from run_memory.db
  → copies to output/Tableau de suivi de visa DD_MM_YYYY.xlsx
```

This chain is ACTIVE_PROTECTED in all current and future steps.

### 9.3 Run Number Assignment

Run numbers are assigned sequentially by `run_memory.db`. The next run number is always `max(existing_run_number) + 1`. `run_0000` is permanently reserved for the baseline.
