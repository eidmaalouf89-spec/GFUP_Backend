## Green-light checklist

### 1. Cache schema versioning landed

**Status:** ✅ **PASS**

`CACHE_SCHEMA_VERSION = "v1"` constant added in `src/reporting/data_loader.py:87`. The constant is written into `cache_meta.json` by `_save_flat_normalized_cache` (`data_loader.py:138-145`) and checked by `_flat_cache_is_fresh` (`data_loader.py:108-118`). The freshness check now rejects any cache that:

- has stale mtime (xlsx newer than cache files), OR
- has missing `cache_schema_version` key, OR
- has a `cache_schema_version` value different from `"v1"`.

**Production confirmation observed during canary:** the very first run after pandas was upgraded showed `[FLAT_CACHE] Cache load failed (non-fatal): (<StringDtype(storage='python', na_value=nan)>, …)` despite the mtime check declaring the cache fresh. With D-001 in place, this exact failure mode is preempted: the cache is rejected before any unpickle attempt. See `docs/audit/CANARY_BEN.md` §13 / D-104, `context/11_TOOLING_HAZARDS.md` H-2.

**Verification protocol (one-line probe):**
```python
# Bumping CACHE_SCHEMA_VERSION = "v2" forces a rebuild on next load.
# Restoring "v1" makes subsequent loads hit the freshly-written cache.
```

**Operational note for next pipeline run:** the FLAT_GED cache files on disk predate the D-001 patch and lack the `cache_schema_version` key. The next `load_run_context` call will reject them and rebuild from `FLAT_GED.xlsx` (~30 s one-time cost), then write a new cache with `cache_schema_version: "v1"`. Expected behavior, not a regression.

### 3. Raw↔flat SAS REF gap addressed

**Status:** ⚠ **DEFERRED-with-rationale**

The operator's claim of "~184 SAS REF events for SNI" was reported during Phase 7 visual smoke. Phase 0 §3 forbids reading raw GED at audit-script time — the audit reads `output/intermediate/FLAT_GED.xlsx::GED_OPERATIONS` (which is what the pipeline produces from the raw GED) and compares it forward through B1 → C3 → C5 → E2.

**What Phase 0 verified:**
- Flat-side count for SNI: B1 = 52, C3 = 52, C5 = 0 (per H-3 by design), E2 = 52. Internally consistent.
- The "184 raw" claim was never directly audited because we never opened `input/GED_export.xlsx` ourselves — that would require a separate raw-vs-flat parity harness.

**What's deferred:**
- Open a follow-up ticket "Raw vs Flat SAS REF gap audit (SNI ~184 vs 52)" attached to TRIAGE.md row D-003.
- The audit must reproduce the operator's count protocol on the raw GED workbook (likely counting RAPPEL cells or pre-2026 entries that the SAS RAPPEL filter excludes via `stage_read_flat._apply_sas_filter_flat` lines 202-274).

**Why DEFERRED rather than FAIL:**
- The flat-side counts converge (B1↔C3↔E2 all show 52 for SNI).
- The flat-side numbers are what the UI consumes; the corrupt-cache + workflow_engine-filter compounding bug that hid this from the user (Phase 7 12a-fix2) is now resolved.
- The discrepancy is a SCOPE definition mismatch (raw cell count vs flat status_clean='REF' count), not a data corruption.
- Closing the gap requires reading raw GED, which is upstream-rework scope. See `docs/audit/TRIAGE.md` D-003 and `context/07_OPEN_ITEMS.md`.

**Documentation:** D-003 entered TRIAGE.md classified as `upstream rework` and surfaced in `context/07_OPEN_ITEMS.md` "Open (handed forward)" section.
