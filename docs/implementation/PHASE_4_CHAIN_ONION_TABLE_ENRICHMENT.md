# Phase 4 — Chain+Onion Priority Table Enrichment (Émetteur + Titre)

This MD is **self-contained**. An agent assigned only this phase can execute it cold without reading any other file in `docs/implementation/`.

---

## 1. Objective

Add **émetteur** (canonical company name) and **titre** to every row of the Chain+Onion priority table on the Overview dashboard, sourced from a backend-enriched artifact, with **zero UI computation**.

The change has three parts:

1. **Backend artifact enrichment** — `src/chain_onion/exporter.py` writes `top_issues.json` with two new fields per issue: `emetteur_code`, `emetteur_name` (canonical, e.g. "Bentin"), and `titre`.
2. **Adapter pass-through** — `app.py:get_chain_onion_intel` returns the new fields untouched (still applies the French translation from Phase 2 if installed).
3. **UI table extension** — `ChainOnionPanel` in `ui/jansa/overview.jsx` adds two columns (Émetteur, Titre) to the row grid.

Because `top_issues.json` is regenerated by the chain_onion run (not the main pipeline), this phase **requires re-running chain_onion** to refresh the artifact. The main GED pipeline does NOT need to re-run.

## 7. Plan

### Step 1 — Locate the export hook for top_issues
In `src/chain_onion/exporter.py`, find the function that writes `top_issues.json` (search for `top_issues`). Identify the data structure being serialized — usually a list of dicts derived from `ONION_SCORES` joined with `CHAIN_REGISTER` / `CHAIN_NARRATIVES`.

### Step 2 — Make `dernier_df` (or equivalent) accessible
The exporter already has access to chain-level dataframes. Check whether it also receives the GED `dernier_df` or operations dataframe. Allowed paths, in order:

- **Option A (preferred):** the exporter is already fed an operations / dernier dataframe (in-memory). Use that. Cheap, no signature change.
- **Option B (cleanest if A unavailable):** pass `dernier_df` (or a slim `numero → {emetteur, titre}` map) into the exporter from the orchestrator that calls it. Requires touching the orchestrator. Acceptable, scoped change.
- **Option C (canonical CSV/parquet only):** read from a canonical CSV/parquet artifact already produced by the pipeline (e.g. a chain_onion intermediate or `dernier`-equivalent CSV in `output/`). Acceptable if such a canonical artifact exists.

**Forbidden:** reading `output/intermediate/FLAT_GED.xlsx` (or any `.xlsx`) from inside the exporter. Excel is a heavyweight, slow, schema-fragile read path. The exporter must enrich from in-memory `dernier_df` or a canonical CSV/parquet source — never Excel. If neither in-memory nor a canonical CSV/parquet path is available, **stop and escalate**; do not silently fall back to reading FLAT_GED.xlsx.

Whichever path, build a lookup:
```python
numero_meta = (
    df_ops.drop_duplicates("numero")
          .set_index("numero")[["emetteur", "titre"]]
          .to_dict(orient="index")
)
```

### Step 3 — Add canonical name resolution
At the top of the function (or import once at module scope):

```python
try:
    from reporting.contractor_fiche import resolve_emetteur_name
except ImportError:
    # fallback: read the map directly from consultant_fiche
    from reporting.consultant_fiche import _CONTRACTOR_DISPLAY_NAMES as _MAP  # name TBD; locate the actual symbol
    def resolve_emetteur_name(code):
        entry = _MAP.get((code or "").strip().upper())
        return entry["name"] if entry else (code or "")
```

If neither symbol exists, hard-code the lookup using a literal copy of the small map. Mark it with a `# TODO: centralize after Phase 5` comment. Phase 5 consolidates this.

### Step 4 — Enrich each issue dict
Inside the loop that builds `top_issues`:
```python
numero = str(issue.get("numero", ""))
meta = numero_meta.get(numero, {})
code = (meta.get("emetteur") or "").strip()
issue["emetteur_code"] = code
issue["emetteur_name"] = resolve_emetteur_name(code)
issue["titre"] = meta.get("titre") or ""
```

Append after existing keys. Do not reorder.
